# 风险与可维护性审计

本文档对应“执行 2”，聚焦部署副作用、供应链风险、维护成本和建议改进方向。

## 结论摘要

这个项目的技术方向是清晰的，脚本也具备较强的自解释性，但它属于“高权限、强副作用、低回滚保障”的运维工具。最主要的风险不是语法正确性，而是：

- 对宿主机网络和解析配置的直接改写
- 外部脚本与二进制的在线下载且未严格钉版本
- 自动验证与回滚机制不完整
- 以交互式流程为主，自动化复用成本偏高

## 风险清单

| 严重度 | 风险 | 说明 | 建议 |
| --- | --- | --- | --- |
| 高 | 宿主机网络强副作用 | 安装流程会禁用 IPv6、停用 `systemd-resolved`、重写并锁定 `/etc/resolv.conf`、替换默认路由、写入 iptables/ipset | 在 README 中保留显著警告；后续增加“最小改动模式”或回滚命令 |
| 高 | 外部依赖未钉死 | V2bX 安装脚本来自 GitHub `master`，`ipt2socks` 默认走 latest release API | 改为固定版本号或固定 commit/tag，并在 README 记录升级策略 |
| 高 | 部署失败时回滚有限 | `wg0` 配置失败、V2bX 安装失败或 WARP 配置异常时，没有统一 rollback 步骤 | 增加 `cleanup` 子命令，或至少补充恢复文档 |
| 中 | 交互逻辑难以 CI 化 | 大量 `read </dev/tty>` 设计适合人工执行，不适合无人值守发布 | 增加环境变量 / 参数模式，支持非交互执行 |
| 中 | 模块职责偏大 | `modules/hk-setup.sh` 和 `modules/warp.sh` 都较长，单文件承载安装、验证、持久化和诊断 | 后续可拆成 `lib/` 函数文件或 `modules/*/steps/*.sh` |
| 中 | 配置写入分散 | `/etc`、`/usr/local/bin`、`systemd`、`cron` 多处写入，排错时需要跨多个位置查找 | 保留单一“运行时产物总表”，本仓库已在 `docs/ARCHITECTURE.md` 汇总 |
| 低 | 校验流程偏轻 | 当前只有运行时诊断，没有仓库内静态检查流程 | 使用 `make check` 或 `./scripts/check.sh` 做基础校验 |
| 低 | 历史误操作痕迹 | 本地曾出现字面量目录 `{modules,tools,docs}` | 已清理；并在 `scripts/check.sh` 中加入守护检查 |

## 观察到的优点

- 入口统一，`flyto.sh` 负责分发，认知成本较低。
- 文档、脚本和交互文案基本一致，接手成本比一般运维脚本低。
- WARP 模块通过 `/etc/warp-google/env` 统一端口来源，这一点很利于后期维护。
- 对 WireGuard 路由语义的解释比较充分，说明作者理解核心网络问题。

## 建议的改进顺序

1. 固定外部依赖版本，先降低供应链风险。
2. 补全卸载/回滚文档，明确每一步改动了哪些系统文件。
3. 为安装流程增加非交互参数模式，便于自动化。
4. 逐步把超长模块拆成更小的步骤函数或库文件。
5. 引入 `shellcheck` 和 `shfmt`，形成更稳定的开发检查链路。
